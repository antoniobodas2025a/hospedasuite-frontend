### Guest Payment Flow (Guest → Hotel)
```
Guest checkout:
  CheckoutForm.tsx L103–147
    → creates PENDING booking via createPendingBookingAction()
    → redirects to Wompi checkout URL with:
        public-key = NEXT_PUBLIC_WOMPI_PUBLIC_KEY (platform key, NOT hotel key)
        reference = payment_link.id (NOT booking.id)
    → Guest pays via Wompi
    → Wompi redirects to /book/success?id={payment_link.id}
    → Wompi sends webhook to /api/webhooks/tenant/wompi

Webhook processing (tenant/wompi):
  L22–31: Validates booking reference exists
  L34–42: Fetches hotel's wompi_events_secret for signature verification
  L45–48: Verifies Wompi signature
  L59–127: If APPROVED:
    → Inserts payment into payments table (L75–81)
    → Updates booking status to CONFIRMED (L86–88)
    → Queues QStash email to guest (L98–127)
  L131–141: If VOIDED:
    → Reverts booking to PENDING
```

**CRITICAL FINDING:** The guest checkout uses `NEXT_PUBLIC_WOMPI_PUBLIC_KEY` which is the **platform key**, not the hotel's key. The hotel's `wompi_public_key` is stored in DB but **never used in the guest checkout flow**. This means the platform receives the money, not the hotel directly.

### Subscription Flow (Hotelier → Platform)
```
Onboarding payment:
  PaymentStep.tsx L113–119
    → WompiButton with isSubscription=true
    → Uses NEXT_PUBLIC_WOMPI_PUBLIC_KEY (platform key)
    → Reference format: ONB-{timestamp}
    → Redirects to /software/onboarding/success

Platform webhook (platform/wompi):
  L76–86: Extracts hotelId from reference (billing-{hotelId}-{YYYY-MM})
  L89–188: If APPROVED:
    → Updates hotel subscription_status to 'active'
    → Logs audit event
    → Sends confirmation email to hotel email
    → Inserts billing_invoices record
```

| Key | Storage | Field | Notes |
|-----|---------|-------|-------|
| Platform public key | Environment | `NEXT_PUBLIC_WOMPI_PUBLIC_KEY` | Used for guest checkout AND onboarding |

| Platform events secret | Environment | `WOMPI_PLATFORM_INTEGRITY_SECRET` | Used for platform webhook verification |
| Hotel public key | Database | `hotels.wompi_public_key` | **Set manually via dashboard, NOT onboarding** |

**CRITICAL:** The `wompi_events_secret` field is referenced in `tenant/wompi/route.ts` L36 but does **NOT exist** in the database schema (`types/database.ts` L709–923). The schema only has `wompi_integrity_secret`. This will cause webhook signature verification to fail because it queries a non-existent column.

## 8. GAPS ANALYSIS — PRIORITY ORDERED

### What Prevents a Hotelier from Receiving Their First Payment Within 48h

The first reservation cycle is:
```
Hotelier signs up → Onboarding → Hotel is live → Guest books → Guest pays → Hotel receives money
```

**Current blockers in this flow:**

1. **Onboarding does NOT collect Wompi keys** → Hotelier must discover this need manually
2. **Readiness check requires Wompi keys for Go Live** → Hotelier cannot go live without them
3. **No welcome email explains "next steps"** → Hotelier has no guidance on what to do after onboarding
4. **No guided Wompi setup flow** → Hotelier must navigate dashboard settings blindly
5. **Guest checkout uses PLATFORM key, not hotel key** → Even if hotel configures keys, guests still pay through the platform account

### Severity-Sorted Gap List

| # | Gap | Severity | Area | Impact |
|---|-----|----------|------|--------|
| 1 | **`wompi_events_secret` column missing in DB** | **CRITICAL** | Wompi Webhooks | Guest payment webhook **will crash** because it queries a non-existent column. All guest payments after Wompi approval will fail to update the booking status. |
| 2 | **No Wompi key collection in onboarding** | **CRITICAL** | Onboarding | Hotelier cannot receive guest payments without manually configuring keys post-onboarding. Breaks the "48h to first payment" promise. |
| 3 | **Guest checkout uses platform Wompi key, not hotel key** | **CRITICAL** | Checkout | Money goes to the platform, not the hotel. Requires manual reconciliation. The hotel's `wompi_public_key` is completely unused in the guest flow. |
| 4 | **Demo bypass button visible in production** | **HIGH** | Onboarding | Anyone can skip the onboarding payment entirely. |
| 5 | **No welcome email with next steps** | **HIGH** | Emails | Hotelier is left without guidance. Support tickets and churn increase. |
| 6 | **No "link in bio" optimized page** | **HIGH** | Link Directo | Social media is a major acquisition channel. A full OTA page is not optimal for bio links. |
| 7 | **No guided post-onboarding setup flow** | **HIGH** | Dashboard | Dashboard should immediately show "Complete your setup: Connect Wompi → Go Live" instead of just metrics. |
| 8 | **QR code generator is just a passthrough** | **MEDIUM** | Link Directo | `api/qr/generate` returns the input string. No actual QR image generation. |
| 9 | **No explicit Wompi sandbox toggle** | **MEDIUM** | Wompi | Hardcoded test keys make it easy to accidentally process test payments in production. |
| 10 | **No email notification for manual payment approval** | **MEDIUM** | Emails | Hotelier must check dashboard manually to know their payment was approved. |
| 11 | **No `billing_payments` table** | **LOW** | DB | Commented out in `billing-success/route.ts`. Not blocking but missing audit trail. |
| 12 | **No admin notification when manual payment is pending** | **LOW** | Emails | Admin must proactively check pending payments. |

### Recommended Priority Order for Fixes

#### Phase 1: CRITICAL — Unblock the Payment Flow (Week 1)
1. **Add `wompi_events_secret` column to `hotels` table** (or fix webhook to use `wompi_integrity_secret`)
2. **Add a Wompi key collection step to the onboarding wizard** (Step 8 or integrate into SettingsStep)
3. **Fix guest checkout to use hotel's `wompi_public_key`** instead of platform key
4. **Remove or hide the demo bypass button** in production

#### Phase 2: HIGH — Improve Conversion & UX (Week 2)
5. **Send welcome email after onboarding** with public URL, dashboard link, and "Connect Wompi" CTA
6. **Add a guided "Complete your setup" widget** to the dashboard for new hoteliers
7. **Create a dedicated `/bio/{slug}` page** optimized for social media (mobile-first, one-tap booking)

#### Phase 3: MEDIUM — Polish & Operations (Week 3)
8. **Implement actual QR code generation** (server-side or client-side with `qrcode.react`)
9. **Add explicit Wompi sandbox mode** (environment toggle or runtime switch)
10. **Add email notifications** for manual payment approval/rejection to hotelier

#### Phase 4: LOW — Audit & Monitoring (Week 4)
11. **Create `billing_payments` table** for subscription payment audit trail
12. **Add admin notification** when manual payment is pending
